PID-PARSE / SMARTPLANT P&ID

Publish XML 导出流程与格式约束

本图描述当前仓内离线 publish pipeline：从 SmartPlant 备份得到 MDF，再由 Rust MDF loader 读取表数据，按 drawing UID 生成 _Data.xml 与可选 _Meta.xml。格式说明以 SPPID 参考导出样本、当前 writer 实现、 CLI 守门项共同约束。

DATA XML

_Data.xml 主文档

  • XML 声明固定为 <?xml version ="1.0" encoding="UTF-8"?>。
  • 根为 <Container>，CompSchema="PIDComponent"，Scope="Data"。
  • Plant 来自 CLI --plant；DocUID / DocName 来自 drawing。
  • 首个业务节点为 <PIDDrawing>，随后输出设备、管线、仪表、表示对象与关系。
  • SPPID 对象格式是“PID tag + 多个 I* 接口节点”；接口和属性集合由 parity 测试守门。

META XML

_Meta.xml 配套文档

  • 根同为 <Container>，但 CompSchema="DocVersioningComponent"。
  • 固定三类实体：DocumentVersion、DocumentRevision、File。
  • 固定三类关系：VersionedDoc、RevisedDocument、FileComposition。
  • 内部 UID 由 drawing UID 确定性派生，保证同一 drawing 重复导出稳定。
  • IFile FilePath 可为空；真实 SPPID 可能写操作员本机路径，此项属 field policy。

SERIALIZATION

序列化与兼容

  • 当前 writer 输出 plain UTF-8、两空格级缩进、尾随换行，写文件走 std::fs::write。
  • 属性值必须 XML escape：&、<、引号、CR/LF 均需可往返。
  • 未知可选字段以空属性保留，如 Description=""，匹配 SPPID blank-but-present 风格。
  • 样例层存在差异：A01 为 UTF-8 无 BOM + LF；DWG 样本为 UTF-8 BOM + CRLF。
  • 因此 BOM / CRLF 不宜判作 schema 问题；若要字节级对齐，应显式引入 serialization policy。

CLI

导出命令

  • 列 drawing：cargo run --bin pid_publish_xml -- Export.mdf --list-drawings
  • 只导出 Data：cargo run --bin pid_publish_xml -- Export.mdf --drawing <UID> --plant TEST02 --out A01_Data.xml
  • 同时导出 Meta：cargo run --bin pid_publish_xml -- Export.mdf --drawing <UID> --plant TEST02 --out A01_Data.xml --meta-out A01_Meta.xml
  • DWG 风格：--style dwg；默认 A01 风格：--style a01。

SPPID SHAPE

格式验收口径

  • 结构验收：XML 可解析，根 Container 与 schema 标记正确。
  • 语义验收：PID tag 数量、接口集合、属性集合与参考 publish XML 对齐。
  • 关系验收：<Rel> / <IRel UID1 UID2 DefUID> 覆盖 model item、representation、业务关系。
  • 风格验收：A01 / DWG 的 IObject 命名约定按 PublishStyle 显式选择。
  • CI 守门：--diff-against 输出 tag 与 Rel-DefUID 级 SemanticDiffReport。
