## 字节审计能力（v0.10+ 新增 Phase 12b 系列）

`pid_parse::byte_audit` 提供与语义解析层并行的字节级覆盖追踪。每个被注册
的 `_with_trace` parser 都会把声明消费的字节按 `TraceConfidence` 三级
归类，未被消费的字节直接进入 `leftover`，整包累积成
`ByteAuditReport`。该机制用于 roadmap Phase 4 的覆盖率回归门槛。

### TraceConfidence 三级

| 等级 | 含义 | 典型场景 |
|---|---|---|
| `Decoded` | 字节布局已知 + 字段语义已命名 | OLE PropertySetStream prefix / cluster 16B 头 / DA 31B trailer / `P&IDAttributes` class name run |
| `Probed` | 位置已知但字段语义尚未命名 | PSMcluster0 heuristic locator gap / Sheet text run / JProperties 文本片段 / 未识别的 VT tag (4B 头) |
| `leftover` | 未声明消费 | 二进制 prefix/suffix gap / 未触达的 record body |

### 已注册 trace 流（17 项）

`run_registered_parser` 把以下 stream path 挂上 `_with_trace` 入口：

```
/\x05SummaryInformation                  -> parse_summary_property_set
/\x05DocumentSummaryInformation          -> parse_summary_property_set
/PSMroots                                -> parse_psm_roots
/PSMclustertable                         -> parse_psm_cluster_table
/PSMsegmenttable                         -> parse_psm_segment_table
/PSMcluster0                             -> parse_psm_cluster0
/StyleCluster                            -> parse_cluster_header
/Dynamic Attributes Metadata             -> parse_cluster_header
/Unclustered Dynamic Attributes          -> scan_da_landmarks
/DocVersion2                             -> parse_doc_version2
/DocVersion3                             -> parse_doc_version3
/AppObject                               -> parse_app_object
/JTaggedTxtStgList                       -> parse_tagged_stg_list
/TaggedTxtData/Drawing                   -> parse_drawing_xml
/TaggedTxtData/General                   -> parse_general_xml
/Sheet<N>     (top-level only)           -> probe_sheet_stream + scan_endpoint_records
*/JProperties (任意 storage 下)          -> parse_jproperties
```

未注册的 stream 在报告中以 `parser_name = None` 列出，全量字节落入
`leftover`。这是 Phase 4 的“覆盖轨迹”信号：当未来某个 parser 迁移上来
后，对应 stream 从 unregistered 翻转到 traced，覆盖率明显跳变即可作为
回归 gate 抓取。

### CLI 入口

```
pid_inspect <file.pid> --byte-audit
pid_inspect <file.pid> --byte-audit --byte-audit-baseline <audit.json>
```

`--byte-audit-baseline` 把当前 report 与给定 baseline JSON 做差分；
CI 可将 baseline 提交到 `docs/baselines/*.byte-audit.json`，配合
`.github/scripts/check-byte-audit-baselines.sh` 在有 fixture 的环境
里跑回归比较，没有 fixture 时 skip 退出。

## Consumer 快速上手

### Parse（v0.5.3+）

```rust
use pid_parse::PidPackage;
let pkg = PidPackage::from_path("input.pid")?;
// 或 from HTTP body:
// let pkg = PidPackage::from_bytes(&body_bytes)?;
```

### Edit via declarative plan

```rust
use pid_parse::{PidPackage, PidWriter, WritePlan};

let pkg = PidPackage::from_path("input.pid")?;
let plan = WritePlan::from_json(r#"{
  "metadata_updates": {
    "drawing_xml": "<Drawing><DrawingNumber>X-9999</DrawingNumber></Drawing>",
    "summary_updates": { "title": "Q4 Pipeline Review", "author": "Jane Engineer" },
    "summary_deletions": ["comments"]
  }
}"#)?;

// To file
PidWriter::write_to(&pkg, &plan, Path::new("output.pid"))?;
// Or bytes (v0.5.3+)
let bytes = PidWriter::write_to_bytes(&pkg, &plan)?;
```

### Inspect coverage programmatically

```rust
use pid_parse::{PidPackage, inspect::coverage};

let pkg = PidPackage::from_path("input.pid")?;
let report = coverage::coverage_report(&pkg.parsed);
for entry in &report.entries {
    if matches!(entry.status, pid_parse::ParseCoverageStatus::Unknown) {
        println!("still unknown: {} ({} bytes)", entry.name, entry.stream_size.unwrap_or(0));
    }
}
// Or JSON for CI (v0.6.4+)
let json = report.to_json_pretty()?;
```

### Byte-audit programmatically（v0.10+）

```rust
use pid_parse::{PidPackage, byte_audit};

let pkg = PidPackage::from_path("input.pid")?;
let report = byte_audit::aggregate::byte_audit_report(&pkg);

println!(
    "总字节 {} / 已声明消费 {} / leftover {} (覆盖率 {:.2}%)",
    report.total_file_bytes,
    report.overall_consumed,
    report.overall_leftover,
    report.overall_coverage_ratio * 100.0,
);

for path in &report.unregistered_paths {
    println!("未注册 trace 的流: {path}");
}
```
